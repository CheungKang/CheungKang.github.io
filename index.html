转载声明（本文不用于商业目的，只做学习以及整理知识和记录笔记）

UIImage Apple官方文档

An image object may contain a single image or a sequence of images you intend to use in an animation.

这句话说, 一个 image 对象可以包含一个 image 也可以包含 一组有顺序的 image, 这组有顺序的 image 是用来展示动画的.

官方给出了 4 个 image 对象的使用方向.

  • 使用 UIImageView 在 interface 上展示图像;
  • 可以定制一些系统的 controls, 官方给出的有三个: buttons, sliders, segmented controls;
  • 直接给 view 或者 other graphics context 绘制图像;
  • 把图像传递给其他的 APIs 可能需要 图像数据;

官方给出的 3 种创建方式

  • imageNamed:inBundle:compatibleWithTraitCollection: 或者 imageNamed: 这两个方法, 官方解释: 从 image asset 或者 boundle 里创建一个 image 对象, 会自动缓存图像数据, 对于频繁使用的图像资源, 应该使用此种方式创建.
  • imageWithContentsOfFile: 或者 initWithContentsOfFile: 这两个方法, 官方解释: 创建 image 对象的图像资源并没有在工程的 boundle 里面, 所有每次创建都会从 disk 加载图像资源, 不建议使用此方式重复创建相同的 image 对象.
  • animatedImageWithImages:duration: 或者 animatedImageNamed:duration: 这两个方法, 官方解释:创建一个 image 对象包含一组有序的 image 对象, 用 UIImageView 呈现动画效果.

还有一些其它的方法是用来针对一些特殊的资源, 来创建 image 对象.

官方列举: Core Graphics 或者开发者自己创建的图像资源. 在 UIKit 框架里也可以通过 UIGraphicsGetImageFromCurrentImageContext 方法, 从 bitmap-based 图形上下文中捕获 image.

官方解释: image 对象是不可变的, 它会根据伴随的图像文件或者图像资源的元数据自动的为 image 对象的大多数属性赋值, 并且这些属性不能被改变. 这种不可变的特性, 也就意味着在任何线程中创建和使用 image 对象都是安全的.

需要知道的几个名词

  • image asset: 管理 images 资源的方式
  • assets library: 每一个 Xcode 项目工程里都包含一个资源库, 可以添加重复的 image set
  • image set: 包含一个可以发生变异的 image 对象, 可以为不同的平台, 不同的特性环境, 不同的比例因素提供不同版本的 image 对象.

除了从 disk 加载 image 对象, 还可以通过 UIImagePickerController 从 camera (照相机) 和 photo library (照片库) 获取. 在使用这种方式时, 需要获取用户的权限.

FOUNDATION_EXPORT 与 #define

在 iOS 中,上面两种方式都是用来定义常量的.

  • FOUNDATION_EXPORT

    在 .h 里面
    FOUNDATION_EXPORT NSString * const kMyConstantString;

    在 .m 里面
    NSString * const kMyConstantString = @"Hello";

  • #define kMyConstantString @"Hello"

第一种方式, 在进行对比字符串的时候速度更快, 可以直接使用 (stringInstance == kMyConstantString) 进行比较 (直接比较指针) , 而使用 #define 只能用 ([stringInstance isEqualToString:kMyConstantString]) 来比较(这种是比较每一个字符串是否相等).

NS_ASSUME_NONNULL_BEGIN 和 NS_ASSUME_NONNULL_END

Nonnull区域设置(Audited Regions)

为了避免重复的去需要设置每个属性或者每个方法都去指定 nonnull 和 nullable, 苹果官方提供了两个宏 NS_ASSUME_NONNULL_BEGIN 和NS_ASSUME_NONNULL_END

NS_ASSUME_NONNULL_BEGIN

@interface TestNullabilityClass () 

@property (nonatomic, copy) NSString *sex; 

- (id)itemWithName:(nullable NSString *)name; 

@end 

NS_ASSUME_NONNULL_END

上面被 NS_ASSUME_NONNULL_BEGIN 和 NS_ASSUME_NONNULL_END 包裹的代码中, sex 属性默认是 nonnull 的, itemWithName: 方法的返回值也是 nonnull ，而参数是指定为 nullable 的。

为了更加安全, 苹果官方制定了几条规则:

  • typedef定义的类型的nullability特性通常依赖于上下文，即使是在Audited Regions中，也不能假定它为nonnull。
  • 复杂的指针类型(如id )必须显示去指定是nonnull还是nullable。例如，指定一个指向nullable对象的nonnull指针，可以使用”__nullable id __nonnull”。
  • 我们经常使用的NSError **通常是被假定为一个指向nullable NSError对象的nullable指针。
static函数与普通函数

被 static 关键字修饰的函数叫做:static 函数(静态函数).
static 函数在内存中只有一份, 普通函数在每个被调用中维持一份copy.

函数的定义和声明默认情况下是 extern 的, 但静态函数只是在声明它的文件中可见, 不能被其他的文件所用.

static 函数的好处:

  1. 其它文件中可以定义相同名字的函数, 不会发生冲突
  2. static 函数不能被其他文件所用.

static 函数会被自动分配在一个一直使用的存储区,直到退出应用程序实例, 避免了调用函数时压栈出栈, 速度快很多.

###NSLog打印

最基础的调试方法, 打印日志. 通过打印出的日志来查看程序运行的路径. 系统NSLog打印的信息较少, 效率较低, 常常使用宏来优化系统的打印方法.

swift版
func CKLog<T>(message: T, file: String = #file, method: String = #function, line: Int = #line) {
#if DEBUG
print("<\((file as NSString).lastPathComponent) : \(line)>, \(method)  \(message)")
#endif
}
OC版
#ifdef DEBUG
#define CKLog(fmt, ...) NSLog((@"<%s : %d> %s  " fmt), [[[NSString stringWithUTF8String:__FILE__] lastPathComponent]   UTF8String], __LINE__, __PRETTY_FUNCTION__,  ##__VA_ARGS__);
#else
#define CKLog(...)
#endif

###断点

  • 普通断点

断点(Breakpoint):当程序运行到断点处时会暂停运行.

  • 条件断点

条件断点是在普通断点的基础上添加了判断条件。当程序执行到断点处，并且满足设置的条件时断点才会有效.

在Edit Edit Breakpoint…窗口中有下面四个输入项：

1. Condition 设置的条件。
2. Ignore 表示忽略多少次之后断点有效。例如：不设置条件，Ignore设置为4，则i=4时程序在断点处停止。
3. Action 在程序断点处执行的操作（执行完这个操作后代码停止运行，此时设置断点处的代码还没有执行）。这里的操作是LLDB语句，关于LLDB会在下面介绍。
4. Options 当选中时，执行完Action的操作后代码不会停止，就像没有设置断点一样。
  • 异常断点Exception BreakPoint

在设置异常断点的情况下 当程序crash时 Xcode会帮我们定位到crash产生的位置。

  • 符号断点Symbolic Breakpoint

符号断点能够为某一个方法或者 某一个类的某一个方法设置断点。
